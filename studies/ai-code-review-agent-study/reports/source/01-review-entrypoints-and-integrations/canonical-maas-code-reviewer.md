# Dimension 01: Review Entrypoints & Platform Integrations — canonical-maas-code-reviewer

## Overview

**canonical-maas-code-reviewer** is a CLI tool that reviews GitHub pull requests and Launchpad merge proposals using a Gemini LLM. It is built by Canonical's MAAS team and distributed as both a Python package (via `uv`) and a Snap package.

**Score: 7/10** — Clean CLI design with a well-structured GitHub Actions workflow. Permissions are minimal and security is considered (`pull_request_target` + read-only checkout). However, there is only one automated entrypoint (GitHub Actions), Launchpad requires manual automation setup, and there is no webhook server, GitHub App, or multi-platform CI support.

---

## 1. Supported Entrypoints

| Entrypoint | Type | File Evidence |
|---|---|---|
| GitHub Actions (PR opened) | CI workflow | `.github/workflows/review-pr.yml:1-50` |
| CLI: `review-pr` | Manual/CI invocation | `src/maas_code_reviewer/cli.py:197-253` |
| CLI: `review-mp` | Manual invocation | `src/maas_code_reviewer/cli.py:129-149` |
| CLI: `review-diff` | Local review | `src/maas_code_reviewer/cli.py:152-194` |
| CLI: `list-lp-mps` | Query tool | `src/maas_code_reviewer/cli.py:120-126` |
| Snap distribution | Package install | `snap/snapcraft.yaml:1-39` |

**The primary automated entrypoint is the GitHub Actions workflow** (`.github/workflows/review-pr.yml`). It triggers on `pull_request_target` with `types: [opened]` — meaning it fires only when a PR is first created, not on subsequent pushes.

All other entrypoints are manual CLI commands. There is no webhook server, no GitHub App, no GitLab integration, and no hosted/SaaS deployment.

---

## 2. Design Classification

**Primarily a self-hosted CLI tool with a companion CI workflow.**

- It is NOT a hosted service (no server component, no listener daemon).
- It is NOT a GitHub App (no app registration, no webhook receiver).
- It IS a CLI tool that can be run anywhere `uv` or Snap is available.
- The GitHub Actions workflow wraps the CLI invocation — it runs `maas-code-reviewer review-pr` as a step.

The `review-diff` command is fully portable (takes any unified diff via stdin or file), while `review-pr` and `review-mp` are platform-specific wrappers.

---

## 3. Trigger Flow (GitHub PR)

```
1. PR opened on GitHub
2. GitHub Actions triggers pull_request_target event
3. Workflow checks out PR HEAD (ref: github.event.pull_request.head.sha)
4. Installs maas-code-reviewer via Snap
5. Runs: maas-code-reviewer review-pr \
         --model "$MODEL" \
         --repo-dir "$GITHUB_WORKSPACE" \
         "https://github.com/${{ github.repository }}/pull/${{ github.event.pull_request.number }}"
6. CLI parses PR URL via parse_pr_url() (src/maas_code_reviewer/github_client.py:109-163)
7. Fetches diff via GitHub API (src/maas_code_reviewer/github_client.py:24-54)
8. Sends diff to Gemini LLM for structured review (src/maas_code_reviewer/reviewer.py:87-147)
9. Posts review as PR review via GitHub API (src/maas_code_reviewer/github_client.py:72-106)
```

The Launchpad flow is similar but uses `launchpadlib` to fetch the MP, clones/merges repos using `GitClient`, generates a diff, reviews it, and posts a comment.

## 4. Authentication Model

| Service | Auth Method | Source |
|---|---|---|
| GitHub API | Personal Access Token (`--github-token` or `GITHUB_TOKEN` env var) | `src/maas_code_reviewer/cli.py:201` |
| Gemini LLM | API key (`--gemini-api-key-file` or `GEMINI_API_KEY` env var) | `src/maas_code_reviewer/cli.py:274-284` |
| Launchpad | OAuth via launchpadlib (credentials file `--launchpad-credentials` or `LP_CREDENTIALS_FILE` env var, or interactive browser flow) | `src/maas_code_reviewer/launchpad_client.py:29-35` |

**Key observations:**
- The GitHub token is used directly in the `Authorization: Bearer` header for diff fetching (`src/maas_code_reviewer/github_client.py:49`) and via PyGithub for PR operations (`src/maas_code_reviewer/github_client.py:22`).
- The workflow uses `${{ secrets.GITHUB_TOKEN }}` which is auto-generated by GitHub Actions and scoped to the workflow's `permissions` block.
- The Gemini API key must be stored as a repository secret (`secrets.GEMINI_API_KEY`).
- There is no token rotation, no app-level auth, and no OAuth flow for GitHub (it's a simple PAT).

## 5. Repository Permissions

**In the GitHub Actions workflow** (`.github/workflows/review-pr.yml:10-12`):

```yaml
permissions:
  contents: read
  pull-requests: write
```

- `contents: read` — needed to check out the PR HEAD so the LLM can read files for context.
- `pull-requests: write` — needed to post review comments via `pr.create_review()`.

This is a **minimal permission set** for the task. The `pull_request_target` event is used instead of `pull_request` so the workflow runs in the context of the base branch (has access to secrets) while still being triggerable by fork PRs.

**For manual CLI usage**, the PAT needs at least `repo` scope (for private repos) or `public_repo` (for public repos) plus `pull_request: write`.

## 6. Installation Complexity in a Private Repo

**Estimated time: ~15–30 minutes.**

1. Copy `.github/workflows/review-pr.yml` into the target repo.
2. Create a `GEMINI_API_KEY` secret in the repo settings (get key from Google AI Studio).
3. Optionally change the `MODEL` env var in the workflow.
4. That is it — `GITHUB_TOKEN` is auto-provisioned by GitHub Actions.

**Complexity factors:**
- No GitHub App installation required (no app registration, no permissions approval screen).
- The Snap package is installed from the Snap Store inside the workflow — no build step.
- `pull_request_target` vs `pull_request` is a subtle distinction that could cause confusion.

The workflow references the `edge` Snap channel by default (`.github/workflows/review-pr.yml:25`), which means users get pre-release versions unless they change it to `stable`. This is documented as intentional dogfooding.

## 7. Security & Operational Risks

| Risk | Detail | Severity |
|---|---|---|
| `pull_request_target` misuse | If the workflow ever ran code from the PR checkout, it would expose secrets. The workflow uses checkout only for file context (the LLM reads files), and runs the CLI from a Snap — not from PR code. | Low (correctly mitigated) |
| PAT scope creep | The CLI uses a PAT which may have broader scopes than needed; there's no mechanism to restrict it to `pull_requests: write`. | Medium |
| No review rate limiting | The workflow triggers on every `opened` event. There's no check for existing reviews before posting (done at the Launchpad client level but not at the GitHub Action trigger level, though the CLI itself doesn't deduplicate for GitHub). | Low |
| API key stored in GitHub secrets | Standard practice, no alternative. | Low |
| No webhook signature verification | No webhook server at all, so this is not applicable. | N/A |
| Snap auto-updates | Snap packages auto-update by default, which could introduce breaking changes. | Low |

The `pull_request_target` usage is **correctly handled**: the workflow checks out PR code but never executes it. The `maas-code-reviewer` binary comes from an independently published Snap package, not from the PR's code. This is the recommended pattern for `pull_request_target` workflows.

## 8. Ease of Adaptation for Ultraplan

**Which integration model would be easiest to adapt?**

The `review-diff` command is the most portable component. It accepts a unified diff from stdin or a file and returns a review. This is the cleanest interface to reuse because:

- **No platform dependencies** — it only needs the Gemini API key and a diff.
- **The LLM review engine** (`src/maas_code_reviewer/reviewer.py:87-147`) is fully decoupled from GitHub/Launchpad concerns.
- **`RepoTools`** (`src/maas_code_reviewer/repo_tools.py:8-41`) is a self-contained, security-scoped file access helper that could be reused directly.

**Patterns worth copying:**

1. **CLI-as-core, CI-as-wrapper** — The tool is a standalone CLI. The GitHub Action is a thin wrapper that calls the CLI with environment variables. This means any CI system (GitLab CI, Jenkins, CircleCI) can integrate it by running the same command.
2. **Snap distribution** — Self-contained binary distribution avoids Python environment issues in CI.
3. **`pull_request_target` with read-only checkout** — Secure pattern for accessing secrets on fork PRs.
4. **Structured review output** (`review_schema.py`) — The JSON review format with inline comments is both GitHub API-compatible and portable to other platforms.

**Tradeoffs to note:**

| Aspect | Current Design | Ultraplan Consideration |
|---|---|---|
| Entrypoints | CLI + GitHub Action | Ultraplan could add webhook server, GitHub App, GitLab webhook |
| Auth | PAT + API key + OAuth | GitHub App installation tokens would be more scalable |
| Event trigger | `pull_request_target [opened]` only | Could support `synchronize`, `labeled`, `review_requested` |
| Cross-platform | GitHub + Launchpad only | Engines are abstracted as clients; adding GitLab = new client |
| Deduplication | Only for Launchpad MPs (marker in comment) | Should be built into the trigger layer |
| Local file access | `RepoTools` with path traversal protection | Excellent pattern to copy verbatim |

The main limitation is that there is **no persistent server** — no webhook listener, no daemon mode. For Ultraplan, adding a webhook server (FastAPI/Flask) that receives push/PR events and dispatches to the review engine would be the natural evolution of this architecture.
