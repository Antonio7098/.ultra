## 1. Supported Entrypoints

Based on the document, reviewdog-github-actions supports **two primary entrypoint models** when used with GitHub Actions:

| Entrypoint | Mechanism | Evidence from Document |
|---|---|---|
| **Raw CLI in workflow step** | Download `reviewdog` binary in a `run:` step, pipe linter output via stdin | Blog post "Basic Setup" YAML: `curl -sfL ... install.sh \| sh -s` + `misspell . \| reviewdog -efm=... -reporter=github-pr-check` |
| **Published GitHub Action** | Use a pre-built Docker container action (e.g., `reviewdog/action-misspell@v1`, `reviewdog/action-golangci-lint@v1`) encapsulating reviewdog + linter | Blog post "Public reviewdog GitHub Actions" YAML: `uses: reviewdog/action-misspell@v1` and `uses: reviewdog/action-golangci-lint@v1` |

Both entrypoints are triggered by the same GitHub event: `on: [pull_request]`.

The document also mentions a third, implicit entrypoint — the **reviewdog GitHub App** — as a pre-existing alternative that the new Actions integration replaces ("No need to create a bot account and personal access token anymore and no need to install reviewdog GitHub App too!").

The document does **not** cover CLI-only usage outside of GitHub Actions, the doghouse server, GitLab CI, Bitbucket, or other CI platforms. Those are documented in the main [reviewdog/reviewdog](https://github.com/reviewdog/reviewdog) repo but out of scope for this source.

## 2. Authentication & Permissions Model

| Auth Method | Scope | Evidence from Document |
|---|---|---|
| `${{ secrets.GITHUB_TOKEN }}` / `${{ secrets.github_token }}` | Auto-generated by GitHub Actions; scoped to the repository that triggered the workflow | Blog post "Basic Setup": `REVIEWDOG_GITHUB_API_TOKEN: ${{ secrets.GITHUB_TOKEN }}` and "Public reviewdog GitHub Actions": `github_token: ${{ secrets.github_token }}` |
| `GITHUB_TOKEN` (fork PR limitation) | Read-only for PRs from forked repos; reviewdog falls back to log console output | Blog post "Note on GitHub Actions Integration" item 2: "GITHUB_TOKEN doesn't have write access if Pull Requests is from forked repositories, so reviewdog will report results to GitHub Actions Log console instead of using Check API." |

**Permission requirements (inferred from document context):**
- **GitHub Action with `GITHUB_TOKEN`**: The token is auto-scoped by GitHub to the workflow's repository. For a standard workflow, `GITHUB_TOKEN` has write access to pull requests and checks on the same repo. No explicit permission scopes are configured by the user — they are inherited from the Actions environment.
- **For forked repository PRs**: The token is read-only, so Check API writes silently fail. The document explicitly calls this out as a limitation.

The document emphasizes that the `GITHUB_TOKEN` approach eliminates the need for a bot account, PAT, or GitHub App installation — the major selling point of the integration.

## 3. Platform-Specific Assumptions

### GitHub Actions specificity

The entire document is **GitHub Actions-specific**. Every assumption ties directly to the GitHub Actions environment:

- **Workflow trigger**: `on: [pull_request]` — GitHub Actions event model is assumed
- **Token injection**: `${{ secrets.GITHUB_TOKEN }}` — unique to GitHub Actions (not present in Travis CI, Circle CI, GitLab CI, etc.)
- **Check API as output channel**: `-reporter=github-pr-check` posts results as GitHub Check Runs, which are a GitHub-specific concept
- **Fork PR handling**: The fallback from Check API to log console is based on `GITHUB_TOKEN` permission behavior, which is a GitHub Actions implementation detail
- **Docker container actions**: Published actions (e.g., `reviewdog/action-misspell@v1`) use GitHub's Docker container action format (Dockerfile + `action.yml` metadata)

### API-specific assumptions

- **Check Runs API**: The `github-pr-check` reporter posts to the GitHub Checks API, which requires `checks: write` permission
- **No Rate Limit discussion**: The document does not address GitHub API rate limiting concerns

### Linter-agnostic design

A key design choice visible in the document is that reviewdog remains **linter-agnostic** even within the Actions integration. The action does not ship with a specific linter — instead, the workflow author chooses what to pipe in. This is reflected in:

1. **Raw CLI approach**: `misspell . | reviewdog -efm="%f:%l:%c: %m" -name="misspell" -reporter=github-pr-check` — the user picks the linter and error format
2. **Published action approach**: `reviewdog/action-misspell` bundles `misspell` but the pattern can be replicated for any linter ("If it's common workflow, we can create Public GitHub Actions for more and more languages and tools")

## 4. Operational Tradeoffs

| Dimension | Tradeoff |
|---|---|
| **Setup complexity** | Very low for standard GitHub repos (copy-paste a YAML file). Zero for published actions — just add a `uses:` line. |
| **Fork PR handling** | Major limitation. Open-source repos with external contributors will lose inline review comments on fork PRs. The document acknowledges this as a known constraint of `GITHUB_TOKEN`. |
| **Authentication** | No credentials to manage (beyond what GitHub Actions auto-provides). Eliminates the burden of PAT rotation, bot account management, or GitHub App installation. |
| **Portability** | Zero. The entire document describes a GitHub Actions-specific integration. The same `reviewdog` CLI binary is portable (as documented in the main reviewdog repo), but the `GITHUB_TOKEN`-based auth and Check API reporters are GitHub-only. |
| **Self-hostability** | Excellent. The user manages everything in their own GitHub Actions workflow. No external service dependency. No SaaS required. |
| **CI dependency** | Fully coupled to GitHub Actions. Cannot be used with GitLab CI, Bitbucket Pipelines, Jenkins, or any other CI from the patterns described in this document. |
| **Latency** | Results appear as Check Run annotations during workflow execution. Synchronous from the user's perspective. No external webhook polling or async processing. |
| **Maintenance** | Published actions (Docker containers) need maintenance as reviewdog and linter versions evolve. The document encourages the community to create and share these actions. |

## 5. Answers to Study Questions

### Q1: What are the supported ways to trigger a review?

1. **GitHub Actions `pull_request` event** — workflow runs automatically when a PR is opened or updated
2. **Manual `workflow_dispatch`** (not explicitly mentioned but implied by the YAML structure)

The document only shows `on: [pull_request]` triggers. No webhook, CLI-only, or cron-based triggers are described.

### Q2: Is the tool designed primarily as a hosted service, self-hosted service, CI job, or local CLI?

**Primarily a CI job (GitHub Actions workflow step).** The document describes reviewdog running inside a GitHub Actions workflow, either as a raw `run:` step or through a published Docker container action. The design is embeddable — it becomes a step in the user's existing CI pipeline, not a separate service.

The document contrasts this with reviewdog's other model (GitHub App), stating that the Actions integration eliminates the need for the App. No hosted or self-hosted service component is described in this document.

### Q3: How does it authenticate with the code hosting platform?

Exclusively via **`${{ secrets.GITHUB_TOKEN }}`** (or `${{ secrets.github_token }}` in the published actions). This is an auto-generated, short-lived token that GitHub Actions injects into the workflow environment. No PAT, bot account, OAuth flow, or GitHub App installation is required.

The token is passed to reviewdog via either:
- `REVIEWDOG_GITHUB_API_TOKEN` environment variable (raw CLI approach)
- `github_token` input parameter (published action approach)

### Q4: What repository permissions does it require?

The `GITHUB_TOKEN` permissions are auto-configured by GitHub Actions and are scoped to the repository containing the workflow. By default, `GITHUB_TOKEN` has: `contents: read`, `checks: write`, `pull-requests: write`, `issues: write`, `metadata: read`, and other scopes. The user does not explicitly configure these.

The **effective** permissions needed by reviewdog's `github-pr-check` reporter are `checks: write` + `pull-requests: write` (for Check Runs and PR review comments). The document does not discuss permission scoping or minimization.

### Q5: How hard would it be to install this in a private repo?

**Very easy — under 10 minutes.** The document states: "You can just copy&paste the above configuration to `.github/workflows/reviewdog.yml` and it should work!" Steps:

1. Create `.github/workflows/reviewdog.yml` in the repository
2. Copy the YAML from the document
3. Commit and push
4. Open a PR — reviewdog runs automatically

No external service registration, no token generation, no permission scoping by the user. The `GITHUB_TOKEN` is auto-provisioned.

This is a **significant improvement** over the GitHub App model mentioned in the document, which required installation and separate token retrieval.

### Q6: Does the integration model create security or operational risks?

**Risks identified from the document:**

1. **Fork PR silent failure**: For open-source repos accepting external contributions, reviewdog will silently degrade to log-only output on fork PRs. Users may not notice that inline review comments are missing.

2. **Docker action supply chain**: Published actions (e.g., `reviewdog/action-misspell@v1`) are Docker containers that could theoretically contain malicious code. The document does not mention pinning to SHA digests or verifying action integrity.

3. **Secrets exposure in raw CLI**: The "Basic Setup" example passes `GITHUB_TOKEN` via `env:` block in the workflow YAML. The token is an auto-generated secret, but it is still a credential exposed to any code running in that step.

4. **No permission scoping**: The `GITHUB_TOKEN` has broad default scopes (write to checks, pull requests, issues). The document does not discuss restricting these via GitHub Actions' `permissions:` block to follow principle of least privilege.

5. **No mention of rate limiting**: The Check API has rate limits, but the document does not discuss handling 429 responses or backoff strategies.

### Q7: Which integration model would be easiest to adapt for Ultraplan?

The **GitHub Action embedded-composite pattern** shown in this document is the most relevant for Ultraplan:

1. **Zero-setup auth**: Using `GITHUB_TOKEN` eliminates all credential management overhead. Ultraplan could follow the same pattern — receive the platform's native auth token, no separate signup or API key required.

2. **Step-in-pipeline model**: Reviewdog runs as a single step in a CI workflow, not as a separate service. Ultraplan could offer a composite action (`ultraplan/code-review@v1`) that users add as a step. This is the lowest-friction adoption path.

3. **Linter-agnostic by default**: The document's core insight is that reviewdog does not dictate which linter to use — it accepts piped output from anything. Ultraplan should similarly accept any formatter/linter output, making it CI-ecosystem-agnostic.

4. **The fork PR limitation is instructional**: Ultraplan must decide how to handle permission-limited environments. The document's fallback to log output is pragmatic but degrades UX. A better approach might be to post a single summary comment when inline annotations are unavailable.

**Limitations of this model for Ultraplan:**
- It is GitHub-only. If Ultraplan needs multi-platform support, it should follow the main reviewdog repo's approach (interface-based abstraction) rather than this document's GitHub-specific pattern.
- Composite actions are harder to version and test than standalone CLI binaries. The raw CLI approach (download binary + pipe) is more portable across CI systems.

## 7. Patterns Worth Copying into Ultraplan

1. **Platform-native token passthrough**: Use the CI platform's built-in token (`GITHUB_TOKEN`, `GITLAB_TOKEN`, etc.) instead of requiring users to provision a separate credential. This is the single biggest friction reducer.

2. **Drop-in workflow YAML**: Provide a copy-paste YAML snippet as the primary installation method. If the snippet is 10 lines or fewer, the adoption rate increases dramatically. The document's `uses: reviewdog/action-misspell@v1` with `github_token: ${{ secrets.github_token }}` is the gold standard for simplicity.

3. **Composite action as distribution mechanism**: Wrapping reviewdog + common linters into reusable GitHub Actions (`action-misspell`, `action-golangci-lint`) creates a marketplace of composable review steps. Ultraplan could offer `ultraplan/review-javascript`, `ultraplan/review-python`, etc., each combining a linter set with the review engine.

4. **Explicit fork-PR degradation strategy**: The document reveals that fork PR behavior is a key design constraint. Ultraplan should design for this from day one — decide whether to post summary comments, skip review, or degrade gracefully, and document the behavior clearly.

5. **Extensibility through stdin pipes**: The `linter | reviewdog` pipe model is elegant because it decouples the analysis tool from the review engine. Ultraplan should accept any input format and let users define their own error format, rather than hardcoding specific linter integrations.
